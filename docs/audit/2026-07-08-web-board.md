**Date:** 2026-07-08
**Agent:** kirei-refactor (audit slice)
**Scope:** apps/web/src/components/board (all files and subdirectories)

## Summary

Audit of the `board` feature slice (~175 source files). The board is a major feature implementing a Kanban task board with heavy use of React contexts, hooks, @dnd-kit drag-and-drop, and streaming session transcripts.

**Overall assessment:** The board demonstrates strong architectural discipline. It follows the documented conventions (folder-per-component, sidecar naming, context consumption, no-state-in-body, barrel discipline) with high fidelity. The three scoped contexts (TaskActionsContext, BoardChromeContext, WorktreesContext) are properly consumed rather than re-threaded as props. State lives in `.hooks.ts` files. File sizes are governed by the ratchet with two grandfathered offenders.

**Categories with findings:**
- Dead code: Minimal (1 logged error path worth noting)
- Duplication: Low (picker patterns are similar but intentionally divergent)
- God files: 2 grandfathered (TaskDetail 526, ProviderConfigPanel 422)
- Abstraction: Good (reuse via `useDisclosure`; no over-abstraction)
- Consistency: High adherence to web AGENTS.md
- Best practices: 1 console.error (acceptable for failure path)

## Duplication to Consolidate

### Picker component patterns (KindPicker, WorkModePicker, PermissionModePicker)
**Files:** `KindPicker/KindPicker.tsx:12`, `WorkModePicker/WorkModePicker.tsx:10`, `PermissionModePicker/PermissionModePicker.tsx:14`
**Pattern:** Segmented radio button groups using `rovingKeydown`, similar selected styling (`border-primary/60 bg-primary/[0.1]`), disabled states.
**Assessment:** Not a high-leverage extraction. Each picker has different:
- Layout (Kind: 2/4-col grid; WorkMode: 2-col; Permission: 3-col + Inherit)
- Data source (KIND_OPTIONS, RUN_MODE_OPTIONS, PERMISSION_MODE_OPTIONS + null-Inherit)
- Semantics (compact mode only on KindPicker; Inherit special case on PermissionModePicker)
**Recommendation:** Leave as-is. The duplication is intentional divergence, not copy-paste debt. If a 4th picker appears, reconsider.

### No other 3+ repetition patterns identified
- Context provider patterns are intentionally different (TaskActions/BoardChrome use `createElement` from feature-root `.ts`; TaskStreamContext is a per-frame stream seam inside TaskDetail).
- Action handler pending checks (`pending(action)`) are repeated by design (each control decides its own disabled/busy state).

## What NOT to Refactor

1. **Picker components** — similar patterns are intentionally divergent (different option counts, layouts, special cases like "Inherit").
2. **Grandfathered file sizes** — TaskDetail.tsx (526) and ProviderConfigPanel.tsx (422) are frozen baselines; touching them without shrinking below cap and updating baseline will fail CI.
3. **Context value shapes** — the three board contexts (TaskActions, BoardChrome, Worktrees) are the deliberate seam; their surfaces are locked by `enforce-context-consumption`.
4. **TaskStreamContext.Provider value={sessions}** — intentionally volatile per-frame; the carve-out in `context-value-must-be-memoized` wiring exists for this reason.
5. **Feature-root modules** (`status.ts`, `chrome.ts`, etc.) — correct per web AGENTS.md grammar for pre-existing bare-noun roots.

## Additional Observations

### Context consumption (enforce-context-consumption)
- All board components correctly consume via `useTaskActions()`, `useBoardChrome()`, `useWorktreesContext()` rather than re-declaring context-provided prop names.
- `isActionPending` is intentionally dual-threaded (listed as excluded from registry in eslint.config.mjs) — this is by design.
- `onResume` is intentionally exposed on BoardChromeValue for the breaker banner (SessionHistory reuses the name for a different resume action; registry comment notes this).

### Context value memoization (context-value-must-be-memoized)
- `BoardChromeProvider` and `TaskActionsProvider` receive values from the shell (AppShell), which memoizes them.
- `TaskStreamContext.Provider value={sessions}` is the explicit per-frame stream seam — correct by design.

### No cross-feature imports
- Board only imports from `@/components/ui` and `@/lib/*` (plus relative within board).
- No imports from `@/components/{app,harness,insight,issues,prreview,projects,scorecard,settings,worktree,onboarding,new-project}`.

### State placement
- No `useState`/`useEffect`/`useReducer` found directly in any `*.tsx` component body.
- All state lives in colocated `*.hooks.ts` (or is read from contexts).

### Barrel discipline
- Component `index.ts` barrels export only the component + its public Props type.
- Feature `index.ts` exports only the app-consumed surface (no internal siblings re-exported "just in case").

### File size governance
- Ratchet cap: 400 raw lines (tests/stories/codegen excluded).
- ESLint `max-lines`: 500 for editor feedback.
- Carve-outs for 7 pre-existing offenders at 1400 (not in board after refactor).
- Board offenders are only the 2 documented in baseline.

### Best practice notes
- One `console.error('pr_status failed', err)` at `PrStatusCard/PrStatusCard.hooks.ts:99` — this is an error path for a fetch failure; the UI shows the error inline and keeps last-good status. Acceptable, but could be routed through a toast if product wants.
- Magic numbers are centralized as constants (ENTRY_PAGE_SIZE=60, ACTIVATION_DISTANCE=6, ESTIMATED_CARD_HEIGHT=140, OVERSCAN=6, etc.).
- No `any` types in production board source (the grep hits were in comments or test fixtures).
