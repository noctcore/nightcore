# M4 Contract — "Earned Verified" (task kinds, verification gate, readiness gauntlet)

**Date:** 2026-06-21
**Status:** FROZEN (orchestrator-owned). Extends M1–M3; serde-additive, breaks nothing.

The big realization: **the existing machinery already carries M4.** A task's `status` is
already the phase discriminator (`InProgress` = a build session is running, `Verifying`
= a reviewer session is running), the correlation FIFO already lets one task own a
*sequence* of sessions (build → review → fix → review …), and there is already an engine
preset seam (`packages/engine/src/agent-presets.ts` + `packages/skills`). M4 = (1) name the
kinds, (2) insert a verification phase between a build's `session-completed` and `Done`, (3)
gate the one irreversible action (merge) on a deterministic local gauntlet. **Reuse, don't
reinvent.**

Goal, one line: today `Done` means "the session exited"; after M4 `Done` means "an
independent reviewer read the diff and passed it."

## Scope (this milestone)
- **D1** Task *kind* presets — the enabling primitive. Full impl of `build` + `review`; `research`/`decompose` are *reserved* enum variants (defined, not yet produced — same pattern M1 used for `Ready`/`WaitingApproval`).
- **V1** Post-run verification gate — `Verifying` status, reviewer session over the worktree diff, `PASS | CHANGES_REQUESTED | FAIL`, bounded auto-fix loop.
- **V2** Pre-merge readiness gauntlet — deterministic local CI (detect → typecheck → lint → test, stop at first fail), gates `merge_task`.
- **Gate merge on PASS** — `merge_task` requires `verified == true`.

**Deferred:** D2 depth modes, D3 model routing, D4 recovery hop, V3 multi-lens, V4 reviewer-model policy → **M5**. F1 decomposition, F2 research-before-build, F3 project memory, F4 on-disk artifacts (`review.md` file) → **M6**. The reviewer is read-only, so it CANNOT write `review.md` this milestone — the verdict text lives on `task.review` (a JSON field); the artifact file is M6/F4.

## A. The kind enum and its split ownership

`TaskKind` is the shared contract; **each tier owns one half of a kind's policy.**

```
build (default) | research | review | decompose      // snake_case on the wire
```

- **Rust core owns ORCHESTRATION policy per kind** (`kind.rs`): `allocate_worktree`, `verify_after`, `writes_code`. This is orchestration → stays in Rust.
  - `build`:   worktree=yes, verify_after=yes, writes_code=yes
  - `review`:  worktree=no (runs in the build's existing worktree), verify_after=no, writes_code=no
  - `research`/`decompose` (reserved): worktree=no, verify_after=no, writes_code=no
- **Engine owns AGENT DEFINITION per kind** (`packages/engine` preset, alongside `agent-presets.ts`): the `appendSystemPrompt`, the allowed/denied toolset, and the default permission mode for that kind. Agent identity already lives engine-side; keep it there.

The enum is the only thing both tiers share. Neither tier reaches into the other's half.

### Engine wire change (additive)
Extend `StartSessionCommand` (`packages/contracts/src/commands.ts`) with **one** optional field:
```ts
kind: TaskKindSchema.optional(),   // 'build' | 'research' | 'review' | 'decompose'
```
`startSession` (`packages/engine/src/session-manager.ts`) resolves `command.kind` (default
`build`) → a `KindPreset { appendSystemPrompt?, allowedTools?, disallowedTools?, permissionMode? }`
and threads those into the `SessionRunner` / `Options`:
- thread `appendSystemPrompt` into `Options` (SDK `appendSystemPrompt`),
- thread `allowedTools`/`disallowedTools` into `Options`,
- a kind's `permissionMode` is a *default* — an explicit `command.permissionMode` still wins.

`review` preset: append a reviewer system prompt (see §B), `disallowedTools` = the write set
(`Edit`, `Write`, `NotebookEdit`, `MultiEdit`, and any apply-patch tool), permission mode that
does **not** prompt (verification is unattended; a stray prompt during review fail-closed-denies
in the core). `build` preset = today's behavior (no append, no tool restriction) so M1–M3 runs are
byte-identical. Keep `nightcoreAgents`/`settingSources` untouched.

Back-compat: `kind` absent ⇒ `build` ⇒ identical to today. Add an engine test that a
kind-less `start-session` still runs, and that `kind:'review'` yields the write-denied toolset.

## B. V1 — the verification gate (in `sidecar.rs::handle_event`, the shared terminal path)

Both `run_task` and the auto-loop funnel terminal events through `handle_event`, so the gate
lives there once and covers both. **`status` is the phase discriminator** — no new session-role map.

### State machine
New status: `TaskStatus::Verifying` (snake_case `"verifying"`).

1. **Build completes** (`session-completed` while `status == InProgress`):
   - If the task's kind has `verify_after == false` (or there is no active project / no
     worktree, i.e. nothing to diff): behave exactly as today → `Done`, `finish_run(Succeeded)`.
   - Else **do NOT call `finish_run`** (keep the slot leased, keep the worktree): set
     `status = Verifying`, persist+emit, then dispatch a **reviewer session** via
     `provider.start_session(task_id, <review prompt>, model=<reviewer model>, cwd=<same worktree>, kind="review")`.
     The slot and worktree are held by the task across build→review→fix; only a *true terminal*
     (`Done`/`WaitingApproval`/`Failed`) runs `finish_run`.
   - The build session id is `forget()`-ten as usual; the reviewer correlates to the same task via the FIFO.
2. **Reviewer completes** (`session-completed` while `status == Verifying`): parse the verdict
   (§B verdict format) from the result text; store the full result on `task.review`:
   - **PASS** → `verified = true`, `status = Done`, `summary` kept, `finish_run(Succeeded)` (NOW the worktree cleans up per policy, slot releases, breaker records success).
   - **CHANGES_REQUESTED** →
     - if `fix_attempts < MAX_FIX_ATTEMPTS` (const = **2**): `fix_attempts += 1`,
       `status = InProgress`, dispatch a **fix build session** (`kind="build"`, same worktree,
       prompt = the original task prompt + the reviewer's change list). Its completion re-enters
       step 1 → re-verification. Slot/worktree still held.
     - else (budget exhausted): `status = WaitingApproval`, `verified = false`, keep `task.review`. `finish_run` is **not** called for `WaitingApproval` either — see §B "holding" below.
   - **FAIL** or **unparseable verdict** → `status = WaitingApproval`, `verified = false`. Fail-safe: never silently pass.
3. **Reviewer/fix `session-failed`**:
   - aborted (cancel/breaker) → today's behavior for the task (Failed/aborted), `finish_run(Aborted)`; verification is moot.
   - genuine failure → `status = WaitingApproval` (verification inconclusive), `finish_run` **not** called for the gate path; but the slot/worktree must still be released — see "holding".

### Holding the slot/worktree without `finish_run`
`finish_run` does four things: release slot, forget session, cleanup worktree, feed breaker+kick.
For the **non-`Done` gate terminals** (`WaitingApproval` from FAIL / budget / inconclusive) we
want to release the slot, forget the session, **retain** the worktree (the user will inspect/approve
it), and **not** feed the breaker (a CHANGES_REQUESTED that the agent couldn't fix is not a "broken
setup" — don't trip the breaker on it), then kick. Add a terminal helper for this
(`park_for_approval` / an `Outcome::NeedsApproval`) rather than overloading `finish_run`. A task
sitting in `Verifying` that gets cancelled drains exactly like any in-flight run (existing
`cancel_task` path: it interrupts the live reviewer/fix session → `session-failed (aborted)`).

### Reviewer prompt + verdict format (core-owned string; the agent identity is engine-owned)
The reviewer runs read-only in the worktree. The prompt instructs it to: inspect *all* changes in
this worktree relative to base branch `<base>` — run `git status`, `git diff`, and check untracked
files — judge correctness/completeness against the task, and **end its final message with a single
machine-readable line**: `VERDICT: PASS` | `VERDICT: CHANGES_REQUESTED` | `VERDICT: FAIL`, with the
rationale (and, for CHANGES_REQUESTED, a numbered change list) above it. The core greps the result
for `VERDICT:\s*(PASS|CHANGES_REQUESTED|FAIL)` (last match wins); no token ⇒ FAIL. `<base>` = the
active project's base branch (the same base `merge.rs` merges into).

### New `Task` fields (all serde-additive, `#[serde(default)]`, camelCase)
- `kind: TaskKind` — default `Build`. Set at create from the picker (§D). `TaskPatch` gains `kind`.
- `verified: bool` — default `false`. True only via a PASS. Cleared on a fresh run.
- `review: Option<String>` — the reviewer's verdict text. `None` until a review runs; cleared on a fresh run.
- `fix_attempts: u32` — default `0`. Reset to 0 on a fresh run; bumped per auto-fix.

Extend the existing back-compat test (`m3_fields_default_and_round_trip`) to pin these four
default on a legacy JSON task. Add the `"verifying"` arm to the status round-trip + snake_case tests.

## C. V2 — pre-merge readiness gauntlet (`gauntlet.rs`, deterministic, zero agent cost)

A detector + runner over the task's worktree. **Never invents commands** — it detects the project's
real tooling and runs only what exists, **stopping at the first failure**.

- **Detect** (in the worktree root): `package.json` → read its `scripts` and pick, in order,
  the ones that exist among `typecheck`/`tsc`, `lint`, `test` (run via the project's pm — prefer
  `bun run <script>` when `bun.lock`/`bun.lockb` present, else `npm run`); `Cargo.toml` →
  `cargo check` → `cargo clippy` (if available) → `cargo test`. A project with none ⇒ gauntlet
  trivially passes (nothing to run) and says so.
- **Run** sequentially, **stop at first non-zero exit**. Return a structured result:
  `{ passed: bool, steps: [{ name, command, status: "passed"|"failed"|"skipped", exitCode? }], failedStep? }`.
  Capture tail output of the failing step for the UI (truncate; never log secrets).
- **Gate**: `merge_task` runs the gauntlet first; on failure it does **not** merge — it returns the
  failed-step result for the UI to surface. (`commit_task` is *not* gauntlet-gated — committing in
  the isolated worktree is reversible; only merge is irreversible.)
- New command `run_gauntlet(id) -> GauntletResult` so the UI can run it on demand (Verified column).
- Tests over a temp git repo + a fake `package.json`: a passing script set passes; a failing
  `test` script stops the run and reports `failedStep == "test"` with `lint`/`typecheck` passed.

## D. Gate merge on PASS, and the web surface

- **`merge_task`** (extend `merge.rs`): require `verified == true` **and** a passing gauntlet (§C)
  before merging. If `!verified`, refuse with a message the card surfaces (the user routes through
  the Verifying/approval flow instead). No force, ever (M3 invariant holds).
- **Approval actions for a verification `WaitingApproval`** (distinct from the *plan* approval in
  `plan_approval.rs`, which resolves a parked `ExitPlanMode` permission — a verification-parked task
  has **no** live session, so those commands don't apply). Add:
  - `accept_review(id)` → `verified = true`, `status = Done` (user overrides the reviewer). Worktree retained for commit/merge.
  - `reject_review(id)` → `status = Backlog` (or `Failed`), keep `task.review` for context.
  - `rerun_verification(id)` → re-dispatch a reviewer session against the current worktree (no rebuild). *Optional this milestone; include if cheap.*
- **Board** (`apps/web`, obeys `@nightcore/eslint-plugin` folder-per-component + stories + tests, sole Tauri seam `lib/bridge.ts`):
  - New **Verifying** column between In Progress and Waiting Approval; cards there show a "reviewing" pulse.
  - **Kind picker** on task create/edit (default Build; Build + Research selectable, Review/Decompose may show "coming soon"). Wire `kind` through create/patch.
  - **Review panel** in TaskDetail: render `task.review` + the verdict, with Accept / Reject (/ Rerun) actions for a verification `WaitingApproval`; a `verified` badge on Done/Verified cards.
  - **Verified column**: a "Run checks" action calling `run_gauntlet`; show the gauntlet step results; `Merge` disabled until `verified && gauntlet.passed`.
  - New bridge fns: `acceptReview`, `rejectReview`, `runGauntlet` (+ `rerunVerification` if built); `kind` added to the create/patch payloads. New web Task fields mirror §B exactly.

## E. Guardrails (all phases)
- **Serde-additive**: every new `Task` field defaults; legacy task JSON still loads. Extend the existing pinning tests.
- **Local-first / Claude-only / unattended-safe**: verification adds no network surface; a reviewer that hits an unexpected permission prompt **fail-closed-denies** (it's autonomous). No token/secret ever logged; gauntlet/diff output is surfaced to the UI but never to logs/telemetry.
- **Slot honesty**: one task holds exactly one slot across its whole build→review→fix→verify cycle. Concurrency accounting never double-counts a task.
- **Breaker discipline**: only genuine *build* setup/exec failures feed the breaker. A CHANGES_REQUESTED the agent can't resolve, a FAIL verdict, or a reviewer crash routes to WaitingApproval and does **not** trip the breaker.
- **No behavior change for verify-less paths**: a `build` task with no active project (no worktree to diff) reaches `Done` exactly as in M3. M1–M3 tests stay green (87 cargo / 118 web / 39 plugin / 210 node).
- **MAX_FIX_ATTEMPTS = 2**, a named const; surface "auto-fix budget exhausted" in `task.review` when hit.
