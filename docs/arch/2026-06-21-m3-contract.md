# M3 Contract — Plan-Approval Gate, Interactive Permissions, Commit/Merge

**Date:** 2026-06-21
**Status:** FROZEN (orchestrator-owned). Extends M1/M2; does not break them.

The big realization: the **engine already does the hard part**. `packages/engine/src/permission-layer.ts` implements the SDK `canUseTool` callback (deny/allow lists, prompt-and-await-surface, `permissionMode` plan/acceptEdits/bypass), emits `permission-required`, and resolves on an `approve-permission` SurfaceCommand. The protocol already defines `approve-permission` and `set-permission-mode`. **The sidecar currently auto-denies every `permission-required` (`createSidecar` in `apps/sidecar/src/index.ts`).** M3 = stop auto-denying, relay to the UI, and add commit/merge. Reuse, don't reinvent.

## A. Permission modes (per the persisted `Settings.permissionMode`, global or per-project)
Map our three settings values to the engine when launching a session (`start-session` config / `set-permission-mode`):
- **`auto-accept`** → engine `acceptEdits` (today's effective behavior — edits proceed; deny-list still applies). No UI prompts.
- **`ask`** → engine `default` — `canUseTool` prompts on non-allowlisted tools → relay to UI.
- **`plan`** → engine `plan` — the agent plans and calls `ExitPlanMode` instead of acting.

The coordinator/sidecar must pass the active task's mode when launching. Fail-closed: if the core can't decide, **deny** (never silently allow).

## B. Interactive permissions (`ask`)
1. **Sidecar** (`apps/sidecar/src/index.ts`): REMOVE the unconditional auto-deny. `permission-required` events already reach stdout via the event sink — keep emitting them; just stop sending the immediate `approve-permission{deny}`. The session parks in the engine awaiting a surface decision (this is correct — long waits are fine).
2. **Rust core**: when the sidecar reader sees `permission-required` for a running task, emit **`nc:permission`** `{ taskId, requestId, toolName, input, suggestions? }`. Add command **`respond_permission(taskId, requestId, decision, updatedInput?, message?)`** where `decision: "allow" | "deny"` → core sends an `approve-permission` SurfaceCommand to the sidecar (`{ behavior:"allow", updatedInput }` or `{ behavior:"deny", message }`; note the SDK REQUIRES `updatedInput` on allow — echo the original input when not rewriting). On task cancel / abort / circuit-break, auto-**deny** any parked requests for that task (fail-closed) so sessions never hang.
3. **Frontend**: render parked permission prompts in the **TaskDetail / Logs panel** — tool name + input summary + **Allow / Deny** (Deny may carry a short reason). Wire to `respondPermission`. A running task with a pending prompt should be visually flagged on its card (e.g. a "needs approval" pulse). `nc:permission` consumed in the AppShell composition root; `respondPermission` added to `lib/bridge.ts` (sole Tauri seam).

## C. Plan-approval gate (`plan`) — a specialization of B
When a task runs in `plan` mode, the agent produces a plan and calls **`ExitPlanMode`**, which surfaces as a `permission-required` for the `ExitPlanMode` tool (its `input` carries the plan text).
- The core recognizes `toolName === "ExitPlanMode"`: instead of (or in addition to) a generic prompt, transition the task to **`waiting_approval`**, store the plan (on the task as `plan?: string` and/or in the stream), and emit `nc:task`.
- Commands (these RESOLVE the parked `ExitPlanMode` request):
  - **`approve_task(id)`** → respond `allow` to the parked request AND `set-permission-mode` → `acceptEdits`, so the SAME session proceeds to build the approved plan. Task → `in_progress`.
  - **`reject_task(id)`** → respond `deny` (with a message); task → `backlog` (or `failed`), session ends.
  - **`refine_task(id)`** → respond `deny`; task → `backlog` with the plan retained in its description/notes for the user to edit and re-run.
- Wire the **Waiting Approval** column actions: the design's `Refine` → `refine_task`, `Commit`/Approve → `approve_task`; add a Reject path. Show the plan in the detail panel.
- A task only enters plan flow when its mode is `plan`; in `ask`/`auto-accept` `ExitPlanMode` won't be the gate.

## D. Commit / Merge (verified tasks)
Git ops in the Rust worktree module, confined to the task's worktree + project (never the user's main checkout; never force).
- **`commit_task(id)`** → in the task's worktree (`nc/<taskId>`), `git add -A` + commit with a message derived from the task title (skip if nothing to commit; surface that). Sets a committed marker.
- **`merge_task(id)`** → merge `nc/<taskId>` into the project's base branch; on success, honor `Settings.cleanupWorktrees` (remove the worktree, keep or delete the branch per policy) and mark the task merged; on conflict, DO NOT force — surface a conflict error/state the UI shows. Emits `nc:task`.
- **Frontend**: wire the **Verified** column actions — `Commit` → `commit_task`, then `Commit`/`Merge` → `merge_task`; reflect committed/merged/conflict state on the card (the design already has `Merged` disabled + a commit branch icon). `commitTask`/`mergeTask` in `lib/bridge.ts`.
- Add Rust tests over a temp git repo: commit produces a commit on the branch; merge fast-forwards/merges into base; conflict is reported not forced; cleanup honors the setting.

## E. Out of scope / later
Per-task model+effort+mode overrides UI, plan diff preview, multi-step approval history, auto-merge policy beyond the single toggle.
