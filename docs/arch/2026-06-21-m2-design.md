# M2 Design — Autonomy + Isolation (the AutoMaker core)

**Date:** 2026-06-21
**Status:** DESIGN (not implemented). Scaffolded seams only — see "Scaffolded seams".
**Builds on:** [`2026-06-21-m1-contract.md`](./2026-06-21-m1-contract.md) (FROZEN),
[`2026-06-21-nightcore-studio-architecture.md`](./2026-06-21-nightcore-studio-architecture.md),
[`../research/2026-06-21-automaker-port-analysis.md`](../research/2026-06-21-automaker-port-analysis.md).

M1 gives a persistent task registry and a Kanban board that runs **one** task at a
time, in cwd, driven manually (click Run). M2 is the layer that makes the board
**autonomous and parallel**: pull eligible tasks off the board, run N at once in
isolated git worktrees, respect dependency ordering, and stop the bleeding when
runs keep failing. This is the AutoMaker crown jewels, re-implemented natively in
the Rust core with hard boundaries instead of AutoMaker's ~60-service Node daemon.

This doc specifies five subsystems the **Rust core** owns, plus the **provider
seam** at the sidecar boundary. It is deliberately a design + clean seams, not a
working loop. Everything below lands in `apps/desktop/src-tauri/src/`.

---

## 0. Principles carried from M1

- **The boundary holds.** The webview drives everything through Tauri commands and
  renders `nc:task` / `nc:session` events. M2 adds commands (`start_auto_loop`,
  `stop_auto_loop`, `set_max_concurrency`) and events (`nc:loop`), but never lets
  the UI reach into orchestrator state directly.
- **The sidecar stays dumb.** All scheduling, concurrency, worktrees, and
  dependency logic live in Rust. The sidecar still only speaks
  `SurfaceCommand`/`NightcoreEvent`. Provider selection happens inside the engine
  provider factory behind the existing Bun bridge (D-009) — never an
  `if provider == …` branch in the core.
- **Don't port the long tail.** AutoMaker has ~10 git services and a plan-approval
  state machine. M2 ports the Tier-1 five and nothing else; plan-approval,
  pipelines, auto-merge, and notifications are M3 (Tier 2).
- **Serial guard generalizes to a slot count.** M1's "a task is already running"
  guard (`Sidecar::active`) is `max_concurrency == 1`. M2 replaces the single
  active-task `Mutex<Option<String>>` with a `SlotManager` holding ≤ N slots.

## 2. Auto-loop coordinator

**Owns:** the scan-and-dispatch loop that drains the board.

**State** (managed Tauri state, alongside `TaskStore` / `Sidecar`):

```rust
struct AutoLoop {
    running: AtomicBool,          // started/stopped via commands
    handle: Mutex<Option<JoinHandle<()>>>, // the spawned tick task
}
```

**Tick algorithm** (runs on a `tokio` interval, e.g. 500ms–2s, and also kicked
immediately whenever a run terminates so the board drains without waiting a full
interval):

1. If `!running` or the circuit breaker is `paused`, do nothing.
2. Ask the `SlotManager` how many free slots exist (`free = max - leased`). If 0,
   return.
3. From the `TaskStore`, collect **eligible** tasks: status is `Ready` (or
   `Backlog`, if the loop auto-promotes — a config choice), AND
   `dependency_resolver::deps_satisfied(task, &all)` is true, AND the task is not
   already leased. Order deterministically (e.g. by `created_at`) for reproducible
   runs.
4. Take up to `free` of them. For each: lease a slot, allocate a worktree, mark
   `InProgress` (persist + `nc:task`), and dispatch the run through the provider.
5. On a terminal event for a run (handled by the existing reader path, generalized
   to look up the task by **session id → task id** instead of a single
   `active_task`): persist `Done`/`Failed`, release the slot, schedule worktree
   cleanup, feed the circuit breaker, and kick a re-tick.

**Termination:** the loop is "drained" when no task is eligible and no slot is
leased. The coordinator emits `nc:loop {state:"drained"}` and idles (stays
subscribed so a newly-created/edited task re-arms it).

**Why a kicked interval, not pure event-driven:** dependencies and circuit-breaker
windows are time-relative; a periodic tick is the simplest correct scanner. The
immediate kick on terminal events keeps latency low without a tight spin.

**Correlation change vs M1:** M1's reader tags every event with the single
`active_task`. M2 needs a `HashMap<u64 /*sessionId*/, String /*taskId*/>` so the
reader can route concurrent sessions' events to the right task. This is the one
real change to the M1 `sidecar.rs` reader; the `start-session` reply's
`sessionId` populates the map (the M1 `session-started`/`session-ready` handling
already captures it — M2 stores it keyed instead of on a single slot).

## 3. Concurrency / slot manager

**Owns:** the cap on parallel runs and the abort handle per run.

```rust
pub struct SlotManager {
    max: AtomicUsize,                 // set via set_max_concurrency command
    leased: Mutex<HashMap<String /*taskId*/, Lease>>,
}

struct Lease {
    task_id: String,
    abort: AbortHandle,               // cancel this run (cancel_task / circuit pause)
}
```

- `try_lease(task_id) -> Option<LeaseGuard>`: returns `None` if `leased.len() >=
  max`. Generalizes M1's `Sidecar::active().is_some()` guard.
- `release(task_id)`: drops the lease, freeing a slot; idempotent.
- `free_slots() -> usize`: `max - leased.len()` (saturating).
- `abort_all()`: used by the circuit breaker pause and `stop_auto_loop`.

**Keep it simple (per the AutoMaker analysis):** AutoMaker's lease *reference
counting* exists because of nested resume→execute dispatch. Nightcore has no
nested dispatch in M2, so a flat `HashMap` of leases is sufficient — do **not**
port the ref-counting until there's a real nested-dispatch case.

**M1 → M2 migration:** `Sidecar.active_task: Mutex<Option<String>>` is replaced by
`SlotManager` with `max = 1` reproducing M1 behavior exactly, then `max` is made
configurable. The M1 unit test `serial_guard_blocks_when_a_task_is_active` becomes
the `max == 1` case of `try_lease`.

## 4. Git worktree isolation

**Owns:** a per-task branch + worktree so N agents edit disjoint working trees.

**Why:** real parallelism is unsafe in one working dir — two agents would clobber
each other's edits. AutoMaker solves this with `git worktree`; we port the *idea*,
not its ~10 services.

**Lifecycle (per task run):**

| Phase | Action |
|---|---|
| allocate | `git worktree add <project>/.nightcore/worktrees/<taskId> -b nc/<taskId>` off the base branch (e.g. `main`/`HEAD`). |
| run | the provider runs with `cwd = <worktree dir>` (sidecar's `start-session.cwd`). |
| settle | on `Done`: leave the branch for the user to review/merge (M2 does **not** auto-merge — that's Tier 2). On `Failed`/cancel: branch kept for inspection. |
| cleanup | `git worktree remove` per the cleanup policy below; the `nc/<taskId>` branch is retained unless the task is deleted. |

**Cleanup policy:**
- `Done` with successful review (M3 merge) → remove worktree, optionally delete
  branch after merge.
- `Failed` / cancelled → **retain** worktree + branch by default (debuggability);
  a `prune_worktrees` maintenance command (and startup reconciliation) removes
  orphans whose task no longer exists.
- On startup, reconcile: `git worktree list` vs. the `TaskStore`; prune worktrees
  with no live task; `git worktree prune` to clear stale admin files.

**Risk (called out in the architecture doc and analysis):** worktrees are the
highest-risk port. Constraints to keep it small:
- Only `add` / `remove` / `list` / `prune` — no merge/rebase/conflict services in
  M2 (those are M3 auto-merge, Tier 2).
- All worktrees under `<project>/.nightcore/worktrees/` (already gitignored), one
  flat dir per task id. No file-copy-of-config step in M2 (AutoMaker copies
  configured files; defer until a concrete need).
- Base-branch dirty state: if the base working tree is dirty, M2 refuses to start
  the loop with a clear error rather than branching off uncommitted work.

**`cwd` already exists in the contract.** `StartSessionCommand` has an optional
`cwd` (see `packages/contracts/src/commands.ts`). M2 sets it to the worktree path;
no contract change needed. This is the seam that makes worktree isolation a
core-only concern — the sidecar just runs where it's told.

## 5. Dependency ordering

**Owns:** "run a task only when its `dependencies` are all `done`."

`Task.dependencies: Vec<String>` already exists (stored in M1, enforced in M2 —
exactly as the M1 contract says). A pure function, trivially unit-testable:

```rust
pub fn deps_satisfied(task: &Task, by_id: &HashMap<String, &Task>) -> bool {
    task.dependencies.iter().all(|dep| {
        matches!(by_id.get(dep).map(|t| t.status), Some(TaskStatus::Done))
    })
}
```

- A missing dependency id (deleted task) is treated as **unsatisfied** (fail
  closed) — the loop won't run a task whose dep vanished; the UI surfaces it.
- **Cycle safety:** the eligibility scan is a single pass over current statuses,
  so a dependency cycle simply never becomes eligible (no infinite loop, no
  deadlock crash) — it stalls visibly. A `detect_cycles` validation on
  create/update can warn the user, but the loop is already cycle-safe by
  construction.
- No topological sort is needed: the per-tick "are all my deps done?" check
  naturally produces dependency-ordered execution as deps complete.

## 6. Failure circuit-breaker

**Owns:** pause the loop after N consecutive failures so a broken setup (auth gone,
repo wedged) doesn't burn through the whole board.

```rust
pub struct CircuitBreaker {
    threshold: usize,                 // e.g. 3 consecutive failures
    window: Duration,                 // e.g. 60s
    failures: Mutex<VecDeque<Instant>>, // timestamps of recent failures
    paused: AtomicBool,
}
```

- `record_success()`: clears the failure window (consecutive count resets).
- `record_failure()`: pushes `now`, evicts entries older than `window`; if the
  count within the window `>= threshold`, set `paused = true`, `abort_all()` the
  in-flight runs, and emit `nc:loop {state:"paused", reason:"circuit-breaker"}`.
- `is_paused()` gates the coordinator tick (§2 step 1).
- `reset()`: a `resume_auto_loop` command clears `paused` and the window so the
  user can retry after fixing the cause.

This mirrors AutoMaker's `AutoLoopCoordinator` breaker (3 failures / 60s → pause)
but as one small, isolated, testable struct rather than entangled in the loop.

## 7. Provider trait seam (where Claude-now / Codex-later lives)

**The seam is the engine provider boundary behind the sidecar transport** (D-009).
Earlier prose said each provider would be a separate sidecar process; Codex
superseded that by running through the existing Bun bridge and engine factory.
The Rust-side abstraction is a `Provider` trait; M2 ships one transport
implementation, `SidecarProvider`, wrapping the M1 NDJSON child.

```rust
/// A driveable agent backend. One Bun sidecar delegates to the engine provider
/// factory (`claude`, `codex`, …) and speaks the same NDJSON
/// `SurfaceCommand`/`NightcoreEvent` protocol — selected by config, NOT by
/// branching inside the core.
#[async_trait::async_trait]
pub trait Provider: Send + Sync {
    /// Ensure the backend is running (spawn lazily). Idempotent.
    async fn ensure_started(&self) -> Result<(), String>;
    /// Start one run; returns the provider/session id used to correlate events.
    async fn start_session(&self, prompt: String, model: Option<String>, cwd: Option<PathBuf>)
        -> Result<u64, String>;
    /// Best-effort interrupt of a run by session id.
    async fn interrupt(&self, session_id: u64) -> Result<(), String>;
    /// Deny (M2) / decide a pending permission request.
    async fn decide_permission(&self, session_id: u64, request_id: &str, allow: bool)
        -> Result<(), String>;
}
```

- **What stays the same:** `NightcoreEvent` is the normalized event union every
  provider emits; the core never sees a provider-native message. A new provider
  implements the engine provider seam and emits these events — zero core changes.
- **Selection:** a `provider` field in config (`claude` default) picks which
  engine provider the existing sidecar starts. No `match provider` scattered
  through the orchestrator — only the Rust transport factory and the engine
  provider factory know concrete implementations.
- **Why a trait at all if there's one impl today:** it pins the boundary so M3's
  second provider is an additive engine factory arm, and it lets the coordinator
  be tested against a mock `Provider` with no child process —
  the same isolation the M1 sidecar tests get from the stub `SidecarManager`.

**Explicitly NOT ported** (per analysis Tier-3): AutoMaker's `BaseProvider` /
`provider-factory` / `model-resolver` / per-provider usage services. The trait
above is ~4 methods; AutoMaker's provider layer is ~30% of its server surface.

## 8. New commands & events (extends, never edits, the M1 contract)

**Commands** (all `Result<T, String>`, args camelCase):

```
start_auto_loop()                    -> ()    // arm the coordinator
stop_auto_loop()                     -> ()    // disarm + abort in-flight
resume_auto_loop()                   -> ()    // clear a circuit-breaker pause
set_max_concurrency(n: usize)        -> ()    // resize the slot pool
```

The M1 six (`list_tasks`, `create_task`, `update_task`, `delete_task`, `run_task`,
`cancel_task`) are unchanged. `run_task` stays as the manual single-run path
(useful even with the loop). `cancel_task` routes through the slot's `AbortHandle`.

**Events:**

```
"nc:loop"  payload = { state: "running" | "drained" | "paused", reason?: string,
                       maxConcurrency: number, leased: number }
```

`nc:task` and `nc:session` are unchanged; `nc:session` now carries concurrent
tasks' streams (already keyed by `taskId`, so the web side needs no change).

## 9. Scaffolded seams (in this change)

To keep M2 reviewable and the boundaries visible, this change scaffolds **empty,
non-wired** seams — types and skeletons with `TODO(m2)` markers, registered
nowhere, behavior unimplemented. They compile and are excluded from `run()` so M1
behavior is untouched:

- `apps/desktop/src-tauri/src/m2/mod.rs` — module doc + re-exports; not wired into
  `lib.rs` (kept out of the build's command surface until M2 implementation).
- `apps/desktop/src-tauri/src/m2/provider.rs` — the `Provider` trait (§7) with a
  doc-only signature; no impl.
- `apps/desktop/src-tauri/src/m2/slots.rs` — `SlotManager` skeleton (§3): fields +
  method signatures returning `todo!()`-free stubs, plus a unit test asserting the
  `max == 1` reduces to the M1 serial guard once implemented.
- `apps/desktop/src-tauri/src/m2/deps.rs` — `deps_satisfied` (§5). This one is a
  pure function safe to fully implement + unit-test now (no I/O, no Tauri), so it
  ships working as the cheapest Tier-1 port.

The coordinator (§2), worktree manager (§4), and circuit breaker (§6) are
**design-only** in this doc — their structs are specified above but not scaffolded,
because they need real `tokio`/`git`/state wiring that would be more stub than
substance. They are the M2 implementation's first tickets.

## 10. Implementation order (M2 tickets)

1. **`deps_satisfied`** (done as a scaffolded, tested pure fn) — wire into a
   read-only "eligible tasks" query first.
2. **`SlotManager`** replacing `Sidecar.active_task`; prove `run_task` still
   serial at `max = 1` (reuse the M1 guard test).
3. **`Provider` trait + `SidecarProvider`** wrapping today's `sidecar.rs` child;
   no behavior change, just the seam.
4. **Session→task correlation map** in the reader (the one real `sidecar.rs`
   change).
5. **AutoLoopCoordinator** (tick + kick) over the above — the first point the
   board actually drains itself.
6. **CircuitBreaker** feeding the coordinator's pause gate.
7. **Worktree manager** (highest risk, last): `add`/`remove`/`list`/`prune` +
   startup reconciliation, `cwd` per run.

Each ticket is independently testable with the M1 pattern (pure logic out of the
Tauri shell; mock `Provider` / temp git repo), so M2 lands incrementally without a
big-bang merge.
