# Nightcore Scan Features Review — Insight, Scorecard, Harness — and the Harness v2 Plan

*Synthesis of a multi-agent review (adversarially verified findings only where marked), 2026-07-01.*

## 1. TL;DR

All three scan features are individually well-built one-shot pipelines, but the composed Profile→Harden→Lock→Verify loop the spec promised does not close: nothing ever writes `.nightcore/harness.json`, so the Structure-Lock Gauntlet runs zero checks for every real user, and an applied lint plugin lands unwired into any eslint config — the flagship output is inert at the last step. The worst shipped bug family is shared and verified: cancelling (or crashing) a run discards every already-paid-for finding/grade in all three features while the results banner explicitly claims partials are shown, and mid-run Convert/Dismiss actions the UI offers always fail because nothing persists until the terminal event. Insight has a verified prompt-injection laundering path (scanned-repo text → finding fields → verbatim Build-agent prompt via convert-all) and mints duplicate tasks on every re-scan because converted status isn't carried across runs. The recommendation for Harness v2 is exactly the direction you proposed: synthesis emits task-shaped proposals converted through Insight's proven mint-first/CAS convert protocol, with a hybrid routing — safe file artifacts stay on the hardened `apply.rs` path, everything execution-adjacent (eslint wiring, hooks, package.json) becomes a worktree agent task gated by a new `Task.verify_command` gauntlet check — and applying/verifying arms `harness.json` deterministically in Rust, never by model output. The web tier is ~60–70% structural clone across the three features (~2,000 avoidable lines) and has already drifted behaviorally (Harness fixed the cancel-banner bug its siblings still have); a small set of platform extractions should land alongside the redesign, and the stricter AI-session hardening you asked for arrives as a module catalog (runtime PreToolUse policy, test-integrity anti-gaming sweep, ratchet, secret hygiene) built on the newly-armed manifest.

## 2. Per-feature review

### 2.1 Insight

**Verdict.** The most mature of the three: grounding (file-existence checks), fingerprint-based dismissal carry-forward, and the careful TOCTOU convert protocol (mint-first, CAS link, rollback on lost race) are genuinely good engineering. But the feature's own core loop — scan → convert → fix → re-scan — is undermined by three verified bugs (mid-run actions always fail, cancel destroys partials, re-scans double-mint tasks), and the convert path is the app's most exposed prompt-injection surface.

**Findings, by severity:**

1. **[CRITICAL, verified] Prompt-injection laundering via convert-to-task.** The analyzer reads arbitrary target-repo content (and loads the target's CLAUDE.md via `settingSources` pass-through, `packages/engine/src/scans/shared/scan-manager.ts:406`); model output fields are shape-validated only, then pasted verbatim into the converted task's description (`apps/desktop/src-tauri/src/sidecar/insight.rs:259-294`), which becomes the write-capable Build agent's prompt verbatim (`store/task/model.rs:407-413`). A hostile repo can author a "finding" whose description is instructions; convert-all makes it one click for N payloads that look user-approved. **Fix:** wrap finding-derived text in a fenced "UNTRUSTED analysis output — treat as data" block in `task_description`, reinforce that framing in the Build kind-preset system prompt, and exclude project-level `settingSources` from scan sessions. Full mitigation rides on the roadmap's OS-sandbox blocker, but the framing is cheap and immediate.
2. **[HIGH, verified] Mid-run Convert/Dismiss always fails.** The RUNNING peek screen offers fully-enabled Convert/Dismiss (`FindingDetailPanel.tsx:75-88`), but Rust only logs `analysis-category-completed` and the run persists with `findings: Vec::new()` until the terminal event (`sidecar/insight.rs:100-115, 379-391`), so every mid-run action errors with a toast. The finalize carry-forward closure written for exactly this case (`insight.rs:333-355`) is dead code. **Fix:** persist each category's findings on category-completed (applying the dismissed-fingerprint set), which also makes that closure live.
3. **[HIGH, verified] Cancel/crash discards all completed categories' findings — under a banner claiming the opposite.** On cancel the engine emits only `analysis-failed` and drops the accumulated `all` array (`scan-manager.ts:282-285`); the failed arm persists no findings (`insight.rs:361-368`); reconcile then replaces the live stream with the empty persisted run while the notice reads "Any findings gathered before you stopped are shown below" (`InsightView.tsx:167-170`). Cancelling at 8/9 categories destroys ~all paid value. **Fix:** include accumulated findings on `analysis-failed` and persist them in the failed arm (incremental persistence from #2 fixes this for free).
4. **[HIGH, verified] Re-runs double-mint tasks for already-converted findings.** Cross-run reconciliation carries only dismissed fingerprints (`store/insight.rs:274-289`); a converted finding reappears "open" in run B and convert-all creates a second task — on every re-scan, for every unfixed-but-converted finding. **Fix:** carry `{fingerprint → linkedTaskId}` for converted findings; if the linked task exists and isn't Done, stamp the re-discovered finding converted+linked (or at minimum exclude it from convert-all).
5. **[HIGH, verified] Zero cost governance.** `start_analysis` hardcodes maxConcurrency/maxTurns/maxBudget to `None` (`insight.rs:137-139`) despite full contract+engine plumbing; defaults are 6 concurrent sessions × up to 40 turns × 9 categories with a full-session retry doubling failed categories; no Rust-side single-run guard exists (`scan.rs:373-390`) and the always-visible History menu lets a user launch a second concurrent scan mid-run. **Fix:** wire a budget control through the existing command fields, reject `start_analysis` while a run is `running` for the project, and gate History selection during a live run.
6. **Quick wins (medium):** require a short verbatim `evidence` quote in the output contract and substring-verify it in `groundFindings` — the single highest-leverage anti-hallucination upgrade (today, hallucinated line numbers are silently clamped into range, `shared/findings.ts:288-331`); persist per-category outcomes + `failure_reason` so a partially-failed run stops reloading as a clean success (`store/insight.rs:160-161`); warn when Diff scope silently falls back to a full-repo scan on a clean tree (`insight.rs:119-125`, `insight/manager.ts:210-217`).

### 2.2 Scorecard

**Verdict.** Clean implementation of a deliberately simpler model (fresh snapshot per run, no reconciliation), but that simplicity means it cannot answer the one question a scorecard exists for — "did the grade improve after I hardened it?" — and its headline letter grade rests on less verification than it appears to.

1. **[HIGH, verified] Cancel/crash/restart destroys all completed dimension grades, and the UI promises the opposite before wiping them.** Same class as Insight #3: `scan-manager.ts:282-284` drops accumulated readings, the failed arm persists status+error only (`sidecar/scorecard.rs:321-327`), and reconcile replaces the transiently-correct stream with an empty run while `ScorecardView.tsx:129-136` says graded dimensions are shown below. **Fix:** upsert each reading on `scorecard-dimension-completed` (or merge accumulated readings in the failed arm); stop advertising partials the persistence layer throws away.
2. **[HIGH, verified] The headline grade is unverified model assertion.** `coerceGrade` maps ANY off-scale answer ("B+", "PASS", "N/A") to a silent 'C' (`scorecard/readings.ts:34-37`); `groundReading` strips fabricated locations but never adjusts the letter resting on them; the UI renders a "% confidence" badge the output contract never solicits (`presets.ts:157-176` vs `ReadingDetailPanel.tsx:48-53`); the a11y rubric aliases not-applicable to A, inflating backend repos. **Fix:** treat off-scale grades as parse errors (the retry exists), add a real not-applicable grade, delete or wire the confidence field, and inject cheap deterministic anchors (lockfile present, CI workflow exists, test-file count, tsconfig strictness) computed engine-side into the prompt as ground truth.
3. **[MEDIUM] The remediation loop never closes.** No git-revision staleness stamp, no cross-run delta, no per-dimension re-grade, no aggregate headline — the user re-runs all 10 dimensions at full cost and eyeballs two grids from memory (`store/scorecard.rs:151-170`, `ScorecardView.hooks.ts:318-328`). **Fix:** stamp HEAD+dirty per run, support single-dimension re-grade (the per-pass architecture makes this cheap), and render grade movement (C→B) — plus the spec's promised sparkline, which was silently dropped.
4. **[MEDIUM] History browsing during a live run orphans the live run's events and exposes "New run" → accidental concurrent paid runs** (`ScorecardView.hooks.ts:119, 177-182, 269-274`). Same missing backend single-flight as Insight/Harness. **Fix:** track live vs viewed runId separately; reject start while a run is running.
5. **[MEDIUM] Dead knobs and dead code:** all three safety knobs hardcoded `None` (`sidecar/scorecard.rs:95-97`); `hardenSkill` exists in two hand-synced copies of which the engine one has zero consumers and the contract doc comment asserts a field the schema lacks (`scorecard/presets.ts:31` vs `sidecar/scorecard.rs:117-131`, `contracts/scorecard.ts:13`); the minted task's value rests on slash-commands (/security-audit etc.) never validated to exist in the user's environment.
6. **Quick win (low):** `ScorecardRun::is_finalized` requires non-empty readings, so an all-ungraded completed run stays permanently re-finalizable — Harness got the widened guard, Scorecard didn't (`scan.rs:236-239` vs `:253-259`).

### 2.3 Harness

**Verdict.** The scan pipeline and the hardened `apply.rs` write path (symlink/clobber defense, atomic temp+rename, execution-sink denylist) are the right bones. But the feature's value chain breaks at both ends: findings are a dead end (no convert-to-task, unlike Insight), and the synthesized artifacts land inert (unwired plugin, unarmed gauntlet). This is why the v2 redesign in §3 is worth doing — it fixes the shape, not just the bugs.

1. **[CRITICAL, verified, cross-cutting] The Harden→Lock arrow does not exist.** A repo-wide grep for `harness.json` finds only consumers (`workflow/gauntlet_project.rs:37`, `workflow/merge.rs:232`, `verification/handlers.rs:116`) — no writer. The synthesis contract never asks for it (`synthesis.ts:254-300`), and apply never emits it. The Verify stage of Profile→Harden→Lock→Verify is a no-op until a human hand-authors the config; the zero-token gate that justifies the whole investment runs zero checks for every real user. **Fix (§3, phase 6):** arm the manifest deterministically in Rust at apply time — never model-authored.
2. **[HIGH, verified] The applied lint plugin lands inert.** The plugin branch activates precisely when an eslint config already exists (`synthesis.ts:254-294`), but `create` refuses to clobber existing files and `merge-section` is allowlisted to agent-doc basenames only (`apply.rs:57-64`) — so wiring the plugin into `eslint.config.*` is structurally impossible, while the reference prompt tells the model to "extend it, do not duplicate it" (`reference.ts:91-95`). The user applies index.js + rules/ + tests/ and nothing ever executes them. **Fix (§3):** route wiring as an agent task with `verifyCommand` = the lint command, plus a narrowly-allowlisted manifest merge.
3. **[HIGH, verified] Cancel/failure discards all streamed findings and paid cost** — same verified class as the siblings: `events.rs:128-136, 164-176` persist nothing per-lens, `commands.rs:70-87` creates the run empty, reconcile clobbers the live fold, and `HarnessView.tsx:147-161` promises partials over an empty grid. **Fix:** incremental persistence on `harness-category-completed` (the handler already mutates the run for the `synthesizing` flag).
4. **[HIGH, verified] Synthesis failure is invisible and unrecoverable.** The most valuable output — paid for by ~8 lens passes — gets no retry (lenses get one, `scan-manager.ts:345-364`; synthesis returns `{artifacts: [], error}` that `manager.ts:245-252` reduces to a warn), no `synthesisError` on the wire, and the Proposed tab renders "No harness artifacts proposed" — identical to a clean repo. Recovery means re-paying the whole scan even though the deduped findings are persisted. **Fix:** give synthesis the corrective retry, persist `synthesisError`, and add a cheap re-synthesize command that feeds persisted profile+findings back in.
5. **[MEDIUM] Findings have no forward path.** Harness findings — especially `gap` kind, which are literally work items — can only be dismissed (`grep convert apps/web/src/components/harness` → nothing), a dead end inside an app whose premise is a task board. Fixed as the headline of §3.
6. **[MEDIUM] Carry-forward is silently wrong over time.** Fingerprint = sha1(kind|targetPath) with content excluded (`synthesis.ts:306-310`); prior status blindly overwrites (`events.rs:66-72`); nothing ever re-checks that an applied file still exists or matches disk. A better re-synthesized rule inherits `applied` and never resurfaces; a deleted artifact keeps its green badge forever. **Fix:** record a content hash at apply time; hash-mismatch → "update available", missing file → "applied (file removed)".
7. **Quick wins (medium):** extend the execution-sink denylist with same-class peers (`.npmrc`, `.yarnrc.yml`, `mise.toml`, `lefthook.yml`, `.cargo/config.toml`, `bunfig.toml`, `.idea/`) in Rust + the TS mirror with a lockstep test (`apply.rs:32-55`, `synthesis.ts:421-437`); enrich `summarizeFindings` with truncated descriptions + evidence anchors so synthesis stops re-exploring the repo from one-line titles (`synthesis.ts:243-251`); either implement "apply as a set" or delete the contract comment and `dependsOn` instruction that currently produce pure token waste (`contracts/harness.ts:175`, `HarnessProposalList.tsx:17-34`); wire the dead budget knobs (`commands.rs:96-99`).

## 3. Harness v2: Propose-then-Convert

This is the plan. It answers your brief exactly — Harness output becomes task-shaped proposals converted via Insight's proven protocol — and every mechanism it leans on was verified present in the code during review.

**Core loop.** Keep the entire 8-lens scan pipeline. Change what synthesis emits and what the user can do with it: synthesis returns `{artifacts, proposals}` where each proposal is task-shaped — `kind: 'apply-artifacts' | 'agent-task'`, `artifactIds`, `prompt`, `verifyCommand`, `harnessCheck`, `fingerprint`. Safe file writes (docs, lint configs at new paths, plugin bundles) stay on the hardened `apply.rs` path, now with bundle-level apply. Everything execution-adjacent — wiring the plugin into `eslint.config.*`, pre-commit hooks, `package.json` scripts, `.npmrc` — becomes a Build task with `run_mode=worktree`, executed by an agent, gated by the new `Task.verify_command` running as a Structure-Lock check before the paid reviewer, and merged through the existing merge-preview flow. This routing is the key idea: it reaches sink-class hardening targets **without weakening the denylist** — the human gate moves from "confirm a file write" to "review a worktree diff", which is strictly stronger.

**Data model.** `HarnessProposalSchema` + additive `proposals` field on the run and events (both codegens, serde-additive per convention). `StoredProposal` lifecycle in `store/harness/status.rs` with fingerprint carry-forward for **both** dismissed and converted (fixing Insight's duplicate-convert bug in the new path from day one). `Task.source_ref = "harness:<runId>:<fingerprint>"` — one serde-additive field, pinned by a format test — giving the two-way link Insight never had. `Task.verify_command` (serde-additive + ts-rs regen). `applied_sha256` recorded at apply time for drift detection.

**Reuse.** The convert protocol is lifted from `convert_finding_to_task` (`sidecar/insight.rs:180-249`): mint-first so a crash leaves a retryable unlinked proposal, CAS link under the store lock, rollback of the duplicate task on a lost race, heal dead links. Rather than hand-copying it a fourth time (it already exists three times: insight, scorecard, decompose `crud.rs:19-104`), extract `sidecar/convert.rs` and migrate Insight onto it in the same commit — Harness becomes the second consumer of the generalization, not clone #4. Converted tasks are ordinary Build tasks, so they inherit worktrees, the gauntlet-before-reviewer wiring (`verification/handlers.rs:119-126`), and merge review for free.

**What happens to `apply_harness_artifact`.** It stays, unchanged in security posture, and gains exactly two things: (1) a bundle apply (`apply_harness_proposal`) that writes an artifact group in `dependsOn`-topological order behind **one** confirm dialog listing all targets — killing the 2N+2 dialog tax and making `dependsOn` load-bearing; (2) a `write_merge_manifest` mode positively allowlisted to exactly `.nightcore/harness.json`, lenient-merge by check name, atomic temp+rename, covered by the existing symlink/clobber bypass tests. When a plugin bundle is applied or a proposal's task is verified, **Rust deterministically writes the corresponding `harness.json` check entry** — the gate config is never model-authored, and the spec's broken Harden→Lock arrow closes.

**Phasing** (each item a small green commit to main, per repo convention):

**NOW — foundations + the headline ask**
1. *Bug-fix foundation, no new product (~small).* Persist `harness-category-completed` findings and `proposals-ready` artifacts incrementally in `events.rs`; replace the `is_finalized` non-empty-collection heuristic with an explicit flag (test the double-terminal case); add `failure_reason` to the shared `PersistedRun` and map it in all three `streamFromRun` folds. This kills the two worst shipped bugs (cancel destroys paid findings; mid-run dismiss always errors) and the verified Insight/Scorecard cancel→red-banner drift, before any redesign code lands.
2. *Contracts + engine (~small-medium).* `HarnessProposalSchema` + additive run/event fields, both codegens. Synthesis emits `{artifacts, proposals}` with a bare-array fallback, gains the corrective retry the lenses have, and stamps `synthesisError` on the terminal event (rendered as a distinct error state). Read-only proposal cards render. Old runs load with empty proposals — zero risk.
3. *Convert (~medium).* Extract `sidecar/convert.rs`, migrate Insight in the same commit, add `convert_harness_proposal` as the second consumer. `StoredProposal` lifecycle with dismissed+converted fingerprint carry-forward. `source_ref` on Task. ProposalDetailPanel with Convert/Dismiss/Go-to-task + convert-all. **This ships your headline ask.**

**NEXT — machine-checkable verify + close the loop**
4. `Task.verify_command`: run it in the worktree as a Structure-Lock check in `verification/handlers.rs` before `dispatch_reviewer`, feeding failures to the existing bounded fix loop. The ConvertConfirmDialog shows the command verbatim (the human gate on model-authored execution) plus an advisory pattern-lint (curl/base64/pipe-to-sh flags).
5. `apply_harness_proposal` bundle apply (one dialog, topological order, per-file results).
6. `write_merge_manifest` + `applied_sha256`: arming shows the full old-vs-merged manifest diff; verified proposals and applied plugin bundles arm the gauntlet. **The spec's #1 critical closes here.**
7. Drift: hash-verified carry-forward replaces blind `applied` inheritance (mismatch → proposed with a "drifted from applied" note); drift banner + Re-propose in ArtifactDetailPanel.
8. Hygiene, one commit: denylist peer sinks + lockstep test; wire `maxBudgetUsdPerCategory` through `commands.rs` to an Advanced row in RunControls (everything downstream already honors it).

**LATER — stricter AI-session hardening, on the now-armed manifest**
9. Runtime PreToolUse policy (`harness-policy.ts`, sibling of `workspace-confinement.ts`): protectedPaths / denyBashPatterns / secret-read-deny globs loaded from the codegen'd manifest — enforced in Nightcore's seam, never target-repo `.claude/settings.json`, holds under bypassPermissions. Ship with a per-task "intends protected change" override or users will disable it wholesale.
10. Built-in zero-token Rust gauntlet check kinds: diff-budget (breach parks, never hard-fails), test-integrity anti-gaming sweep (`.only`/`.skip`/gutted assertions/new ts-ignore, evidence fed to the fix loop), ratchet never-worse baseline.
11. Expansion lenses + proposal templates: secrets-and-supply-chain (gitleaks config + hook wiring as agent-task), verify-loop focus (propose the verify script + verify-command check), dep-cruiser config (making the gauntlet's already-implemented DependencyCruiser kind reachable for the first time), plugin-wiring agent-task with `verifyCommand` = the lint command.
12. Optional, only if usage shows re-runs: a run-delta banner (new/persisting/resolved by fingerprint set arithmetic — no prior-findings prompt digest; the anchoring risk isn't worth it) and a Scorecard grade-delta chip.

**Explicitly deferred:** a full module-catalog UI and per-module synthesis rewrite (revisit only if the monolithic synthesis pass degrades); a persistent posture-ledger store (drift badges + armed-manifest state cover ~80% without a second source of truth); a generic `replace` write mode (the narrowly-allowlisted manifest merge is safer).

## 4. Hardening-module catalog (Later expansion, ordered by value-for-effort)

Each module: what Harness generates, enforcement tier, prior art.

1. **Verify-command contract** — audit/synthesize one fast machine-checkable done-command (`verify` script + AGENTS.md doc + manifest entry); **gate** — the gate everything else leans on. (Factory.ai agent-readiness; CodeScene agentic patterns.)
2. **Test-integrity anti-gaming sweep** — pure-diff detection of `.only`/`.skip`, gutted assertions, new suppressions, gate-config edits, `--no-verify` in Bash history; **gate** (zero-token, pre-reviewer). (Kinney "Making It Hard to Cheat the Guardrails.")
3. **Protected-paths + bypass-flag denial** — manifest-driven PreToolUse rules blocking Edit/Write to lockfiles/migrations/generated code/the manifest itself and Bash escape hatches; **runtime hook**, enforced in Nightcore's seam. (Claude Code hooks exit-2 semantics; dwarvesf/claude-guardrails.)
4. **Secret hygiene** — `.gitleaks.toml` artifact + read-denial of `.env*`/keys via the hook layer + gitleaks against staged changes in Nightcore's own `commit_task`; **hook + gate**. (gitleaks/TruffleHog; Snyk State of Secrets.)
5. **Diff budget + session flight recorder** — per-task changed-lines/files ceiling (breach parks for triage) over a persisted tool-event ledger; **gate + observability**. (Danger JS bigPRThreshold; agent audit-log practice.)
6. **Strictness ratchet** — baseline snapshot of `any`/ts-ignore/eslint-disable counts + never-worse gauntlet check; agents may reduce debt, never add; **gate**. (Betterer; type-coverage.)
7. **Import-boundary lock** — `.dependency-cruiser.cjs` + eslint-plugin-boundaries derived from the profiler's import graph, human-confirmed; **gate** (uses the existing gauntlet check kind). (Xebia fitness functions; Nx boundaries.)
8. **Agent-context budget compiler** — CLAUDE.md/AGENTS.md compiled against an instruction budget (~150), banned-pattern lint, overflow restructured into satellite docs; **advisory doc + meta-lint gate**. (HumanLayer CLAUDE.md guide; GitHub 2,500-repo AGENTS.md analysis.)
9. **Least-privilege permission manifest** — derived deny→ask→allow ruleset emitted as settings artifact and merged into SDK Options per session; **runtime gate** — the concrete first step of the LOCKED tiered-sandbox decision. (Claude Code permissions docs.)
10. **Changed-lines coverage gate** — coverage restricted to the agent's own diff (~80% threshold), uncovered lines fed to the fix loop; **gate**. (diff-cover.)
11. **Dependency firewall** — install-command interception (allow-from-lockfile), `ignore-scripts`/`save-exact` configs, lockfile-lint gauntlet stage, lockfile-changed → park; **hook + gate**. (Slopsquatting research, USENIX 2025; lockfile-lint.)
12. **Prompt-injection surface scan** — deterministic detectors (Unicode tags, zero-width text, instruction-shaped strings) over repo text pre-flight, flagged paths read-quarantined; **scan + hook**. (HiddenLayer CopyPasta; 84% ASR research.)
13. **Env-var contract** — typed env schema (zod/t3-env style) + `.env.example` sync check; **gate + grounding doc**. (t3-env; Factory readiness.)
14. **Ranked repo map for the Context Pack** — tree-sitter symbol graph, PageRank-ranked, budgeted into every session's pre-flight context; **advisory grounding**. (Aider repomap.)
15. **Sandbox profile artifact** — per-repo devcontainer + Seatbelt/bubblewrap config; **OS containment** — strongest tier, blocked on the tiered-sandbox roadmap item. (Anthropic sandbox-runtime.)
16. **Characterization-test bootstrapper** — golden-master tests for high-fan-in/low-coverage modules before agents touch them; **artifact + pre-flight advisory**. (Feathers, Working Effectively with Legacy Code.)
17. **Mutation-score gate-strength audit** — Stryker incremental on critical paths; scores how much "tests pass" is worth per module, feeds the Scorecard Tests dimension; **advisory that tunes gates**. (Google mutation-at-scale.)
18. **Public-API snapshot lock / AST policy pack / commit discipline** — api-extractor surface reports, ast-grep multi-language rules, lefthook+commitlint bundles; **gates**, lower priority until the manifest ecosystem is proven.

## 5. Cross-cutting platform moves

The engine tier is genuinely shared (one `ScanManager`, 1,217 lines of `scans/shared/`), Rust is half-shared (`scan.rs` + `run_store.rs`), but the web tier is a verified ~60–70% structural clone — and it has **already diverged behaviorally**: Harness's reconcile preserves the cancel reason with a comment naming the exact bug (`HarnessView.hooks.ts:91-105`) that Insight and Scorecard still ship (`InsightView.hooks.ts:80-84`, `ScorecardView.hooks.ts:71-78`). Three highest-leverage refactors, in dependency order:

1. **`failure_reason` on the shared `PersistedRun` + a generic `ScanStream<TItem>`/`foldScan` + `useScanRun` hook in `apps/web/src/lib`** (outside the no-cross-feature-imports boundary), parameterized by event map and per-feature fold. Kills `addUsage` ×3, `streamFromRun` ×3, the ~1,720 lines of parallel view-model hooks, and makes the cancel-banner and reconcile-ordering fixes land once. Pair with a `makeScanBridge(channel, prefix)` factory replacing the three parallel subscribe/list/get/delete/cancel blocks in `bridge.ts:1044-1438`. Do this **before** scan feature #4, which currently costs ~1,100 lines of copy-paste before any feature-specific code.
2. **The shared proposal-to-task convert service** (`sidecar/convert.rs`, §3 phase 3) — the mint-first/CAS/rollback/heal protocol exists as three independent hand-copies (`insight.rs:180-249`, `scorecard.rs:139-212`, `crud.rs:19-104`), one of which keeps its critical subtlety only in a comment. Extract it, migrate Insight, make Harness the second consumer, and pick one bulk-convert home (Rust-side with per-item progress events — keeps Insight's aria-live UX and gives Decompose failure reporting for free).
3. **Schema the un-schema'd notices + shared fingerprint spec.** `finding-converted`/`reading-converted`/`artifact-applied` are raw `json!` blobs outside the contracts package with hand-declared TS shapes (`bridge.ts:1061,1221,1316` admit "not a NightcoreEvent") — move them into contracts so they codegen both ways. And give the fingerprint a shared, typed basis: today Insight keys on file|title, Harness on category|title, and Scorecard passes the dimension in the *file* parameter slot (`scorecard/readings.ts:84`) — three semantics living only in comments, one wrong basis away from "my dismissed findings came back."

**Feature composition (the arrows the spec drew but code never built):** Harness v2 phase 6 closes Harden→Lock (manifest arming). Close Profile→Harden by adding a "Generate enforcement harness" CTA on low Scorecard architecture/code-quality grades that deep-links to HarnessView with the relevant lenses preselected. Close Verify→Profile with single-dimension re-grade on a linked Harden task reaching verified plus a grade-delta chip in DimensionGrid — the data is already persisted, only the rendering is missing. Finally, record the two deliberate spec divergences (Scorecard's fixed-pass grading instead of skill dispatch; the context pack's shrunk sources — which currently lets one click fold an untrusted root CLAUDE.md into the trusted `appendSystemPrompt` channel, worth a diff-preview gate) in a short decision note in `docs/research/`, so future sessions can tell drift from decision.
