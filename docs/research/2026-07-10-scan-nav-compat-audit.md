# Compat audit: what breaks when the scan-view nav keys remap

**Date:** 2026-07-10
**Ticket:** wayfinder #86 (follow-up to `docs/research/2026-07-10-scan-views-rethink.md`)
**Status:** complete (research only — no code changes)

## The question

The stage regroup (Understand / Harden / Enforce / Verify) re-slices the scan views
`insight` / `scorecard` / `harness` (and touches `prreview` / `issuetriage`). Persisted
data holds references that resolve through today's view identities. Which stored
references break, and what compat shim keeps old provenance chips routing?

## Root finding

**There is exactly ONE persisted reference class that touches view identity — the task's
`sourceRef` token — and it is already decoupled from the AppView vocabulary by design.**
The token's first segment is a *feature scheme* (`insight:`, `scorecard:`, `harness:`,
`harness-proposal:`, `pr-review:`, `issue-triage:`) minted Rust-side and mapped to an
AppView only at one web-side chokepoint: the `REGISTRY` in
`apps/web/src/lib/source-ref.ts:22-52`. Nothing anywhere persists an AppView key itself —
not settings, not the run stores, not `.nightcore/`, not localStorage (one cosmetic
exception below). So the remap is safe **iff**:

1. the Rust mint formats stay byte-identical (they are feature identities, not nav
   identities — do not rename them to stage names), and
2. the `REGISTRY`'s `view:` values are retargeted to the new stage AppView keys in the
   same commit that changes the `AppView` union.

No dual-read table, no data migration, no write-path change is needed. TypeScript makes
most of the failure modes compile errors; the one silent failure mode is documented in
§ "What happens today" below.

### 2. `routing.scanTarget?.view` preselect plumbing (in-memory, but its vocabulary IS `AppView`)

Producer (only one): `useRouting.hooks.ts:16` state + `:26-32` `gotoSourceRef` — the value
always comes from `parseSourceRef`, i.e. from the `REGISTRY`.

Consumers:

- Gates in `AppShellViews.tsx` — `preselect={routing.scanTarget?.view === '<key>' ? … : null}`
  at lines **181** (insight), **193** (scorecard), **205** (harness), **217** (prreview),
  **229** (issuetriage); consumed-once via `routing.clearScanTarget`.
- The shared effect `apps/web/src/lib/usePreselectNavigation.ts:42-57` (consume-first →
  `selectRun` → `onOpenItem`).
- Per-family `onOpenItem` branches on `ScanTarget.kind`:
  - `insight/InsightView/InsightView.hooks.ts:128-137` (kind `finding` → `setSelectedId`)
  - `scorecard/ScorecardView/ScorecardView.hooks.ts:229-234` (kind `reading`)
  - `harness/HarnessView/HarnessView.hooks.ts:84-99` — the only kind-branching consumer:
    `proposal` → proposals section + `openProposalById`; else conventions section. This
    branch is exactly the Harden/Enforce split seam.
  - `issues/IssueTriageView/IssueTriageView.hooks.ts:191-197` (run-level, `onOpenItem` no-op)
  - `prreview/prreview-navigation.hooks.ts:207-230` (selects PR from the loaded run)

**Breaks if:** the gate comparisons in AppShellViews aren't rewritten to the new stage
keys, or a merged stage shell (Understand hosting Insight+Scorecard) can't tell which
sub-mode a target belongs to. Note `kind` already disambiguates within a stage —
`finding` (insight) vs `reading` (scorecard) vs `proposal`/`finding` (harness) — but
`insight` and `harness` share the kind `finding`, so a merged shell that hosts both
CANNOT key on `kind` alone (see shim spec: add `family` to `ScanTarget`).

### 6. What happens TODAY when a stored ref points at a nonexistent view/run/item

The chain is graceful at every persisted-data layer:

- **Unknown scheme** (e.g. a token from a future version): `sourceRefLabel` → `null` →
  TaskDetailHeader renders **no chip** (`TaskDetailHeader.tsx:38` gates on
  `provenance !== null`); `parseSourceRef` → `null` → `gotoSourceRef` **no-ops**
  (`useRouting.hooks.ts:28`, tested at `useRouting.hooks.test.tsx:70-77`). No crash, no
  toast — silent degradation by design (`source-ref.ts:19-21` comment).
- **Known scheme, deleted run**: `useScanRun.selectRun` (`apps/web/src/lib/useScanRun.ts:236-241`)
  calls `getRun`, gets `null`, returns without touching the stream; `onOpenItem` then
  selects an id that no item matches → detail panel stays closed. Documented contract:
  "A deleted run/item degrades to the current stream with no panel — never an error"
  (`usePreselectNavigation.ts:5-7`).
- **The ONE silent failure mode a remap can introduce**: an `AppView` union member that
  keeps existing while its render branch in `AppShellViews.tsx` is removed. `setView`
  succeeds, no branch matches, `<main>` renders an empty keyed `m.div` — a **blank
  screen**, no compile error, no runtime error. This happens if the remap deletes the
  `view === 'insight'` branch but leaves `'insight'` in the union (or in the REGISTRY).
  Removing union members instead makes every stale literal a compile error — strictly
  better. **Rule: shrink the union, never orphan a member.**

## The compat shim spec

### Chokepoint: `apps/web/src/lib/source-ref.ts` REGISTRY (it already exists — keep it the only one)

The shim is NOT a new module. The REGISTRY at `source-ref.ts:22-52` is already the single
scheme→(view, kind, label) join point; the remap edits its `view:` values and nothing
else in the persistence path:

```
insight:          view 'insight'   → 'understand'
scorecard:        view 'scorecard' → 'understand'
harness:          view 'harness'   → 'enforce'    (kind 'finding' = convention/gap)
harness-proposal: view 'harness'   → 'harden'     (kind 'proposal')
pr-review:        view 'prreview'  → 'verify' (or stays 'prreview' if PR Review keeps its own destination)
issue-triage:     view 'issuetriage' → unchanged (or 'understand' if folded into intake)
```

(Exact stage targets per the rethink doc's Option C; the mechanism is identical whichever
grouping ships.)

### Required extension: add `family` to `ScanTarget`

Today `ScanTarget` is `{ view, kind, runId, itemId }` (`source-ref.ts:11-16`). After the
merge, `view: 'understand'` alone cannot tell the Understand shell whether to open the
find (Insight) or grade (Scorecard) sub-mode, and `kind: 'finding'` is ambiguous between
Insight and Harness-convention. Add the scheme itself as a discriminator:

```ts
export interface ScanTarget {
  view: AppView;                    // the STAGE destination (new vocabulary)
  family: 'insight' | 'scorecard' | 'harness' | 'prreview' | 'issuetriage'; // the engine/store family (old vocabulary, = scheme owner)
  kind: 'finding' | 'reading' | 'proposal' | 'validation';
  runId: string;
  itemId: string;
}
```

The REGISTRY populates `family` per scheme; the stage shells branch on
`family` (+ `kind` for Harden vs Enforce) to pick the sub-mode and the right per-family
`selectRun`/store. This keeps `usePreselectNavigation` and all per-family
`onOpenItem` handlers unchanged in shape.

### Dual-read vs write migration: dual-read is NOT needed; write migration is FORBIDDEN

- **Writes (Rust mints): freeze them.** The six mint formats are feature identities and
  keep working verbatim under any nav grouping. Changing them (e.g. minting
  `understand:…`) would be the only way to actually break old data — and would force a
  REGISTRY dual-read of old+new schemes forever. Add a comment at each mint site pointing
  at `source-ref.ts` REGISTRY as the pairing invariant.
- **Reads: single-read through the retargeted REGISTRY.** Old and new tasks carry the
  same schemes, so one mapping serves both. No `.nightcore/tasks/*.json` rewrite, no
  version stamp, no migration code.
- **If a mint scheme is ever renamed anyway** (do not): the REGISTRY must keep the old
  scheme as an alias entry — it is a `Record<string, …>` so aliasing is one added line —
  and the old entry must never be garbage-collected while pre-rename task files exist.

### AppShellViews / union mechanics (the loud part)

1. Replace union members in `AppShell.types.ts` (remove `insight`/`scorecard`/`harness`
   if their destinations die; add `understand`/`harden`/`enforce`). Never leave an
   orphaned member (blank-screen mode, §6).
2. Rewrite the five preselect gates in `AppShellViews.tsx:181-229` to the new keys; a
   merged shell receives the target when `scanTarget.view` equals ITS stage key and
   dispatches internally on `scanTarget.family`.
3. `nav.constants.tsx` regroups; `useNavShortcuts` follows automatically (data-driven).
4. `localStorage` key `nc:prreview:list-width` and all `.nightcore/` store dirs stay
   byte-identical.

### Parity lint rule (`tools/lint-meta/rules/scan-family-parity.ts`)

- Nav-key-only remap: **no rule change** (the map keys on web folder + View component).
- If folders/components merge (e.g. `understand/UnderstandView` hosting both hook sets):
  update `SINGLE_RUN_FAMILIES` tuples in the same commit or CI reds via rule (a); a new
  `<stage>/<stage>-stream.ts` requires conscious enrolment via rule (d). Validate with
  `bun run lint` + `bun run lint:meta` (zero violations on a clean tree).
- Recommended hardening (small, optional): add a rule assertion that every `REGISTRY`
  `view:` value is a member of the `AppView` union that has a render branch — i.e. grep
  `AppShellViews.tsx` for `view === '<value>'`. This converts the blank-screen mode into
  a CI failure. (TypeScript already catches union-membership; the render-branch half is
  the uncovered gap.)
